/*-------------------------------Настройка проекта с Docker----------------------*/
docker-compose.yml — это настройка проекта с несколькими контейнерами. то “инструкция”, как собрать и запустить твою мини-инфраструктуру локально (например, базу данных, API, LocalStack и т.п.)

            version: "3.8"                  // Это версия синтаксиса Docker Compose файла.

            services:                     // Здесь описаны сервисы (контейнеры), которые нужно запустить.
В данном случае — только один: localstack.
            localstack:  
                container_name: "${LOCALSTACK_DOCKER_NAME:-localstack-main}"   // Имя контейнера.
Если переменная окружения LOCALSTACK_DOCKER_NAME не задана — используется localstack-main.
                image: localstack/localstack     // Это Docker-образ, из которого создаётся контейнер.
В данном случае — официальный образ LocalStack.
                ports:
                - "127.0.0.1:4566:4566"            # LocalStack Gateway            // основной порт шлюза LocalStack  
                - "127.0.0.1:4510-4559:4510-4559"  # external services port range  // порты для эмулируемых AWS-сервисов
                environment:                // Передаёт переменные окружения внутрь контейнера
                - DEBUG=${DEBUG:-0}         // DEBUG=${DEBUG:-0} — если переменная DEBUG не указана, значение по умолчанию — 0.
                volumes:                    // Монтирует директории:
                - "${LOCALSTACK_VOLUME_DIR:-./volume}:/var/lib/localstack"  // чтобы сохранялись данные между перезапусками
                - "/var/run/docker.sock:/var/run/docker.sock"               // позволяет контейнеру взаимодействовать с Docker-хостом (для создания других контейнеров при необходимости)



/*------------------------------- Как запустить проект с Docker -----------------*/
- после инициализации проекта открывает приложение Docker;
- в комендной строке нашего проекта запускаем команду:
            npm run docker-compose up  может использовать флаг -d = detached mode, то есть: Запустить контейнеры в фоновом режиме, не блокируя терминал.
        или то что будет написанно в package.json:
            "docker-compose:up": "docker compose -f docker/local/docker-compose.yml up -d"
             docker-compose:down - остонавливает контейнер
- запускаем проект согласно скрипну в package.json: "start:watch": "concurrently --kill-others \"npm run start:watch:api\" \"npm run start:watch:web\"",
            npm run start:whatch


/*---------------------------------Осноыные команды---------------------------*/
ss -tuln | grep 4566 - проверить длкальній порт с числом 4566
docker ps - (process status) вывод списка активных запущенных контейнеров
docker-compose ps - выводит список запущенных контейнеров, относящихся к текущему проекту
docker-compose restart - перезапускает контейнер, перезапуск контейнера происходит когда в еминале мы на ходимся в той папке где файл docker-compose.yml







/*-----------------------------Взаемодействие AWS Docker---------------------*/
1️⃣Использкя возможности Docker создадим локальный бакет S3 (AWS) (на который в настоящем AWS у тебя могли быть ограничения).
    aws --endpoint-url=http://localhost:4566 s3 mb s3://my-local-test-bucket
         --endpoint-url — это секретный ключ, который говорит команде: «Не иди в интернет, иди в мой Docker», Если он ответит make_bucket: my-local-test-bucket, значит, ты официально стал «властелином» своего облака.
         http://localhost:4566 - это наш локальный адрес где запушен Docker

2️⃣ Загрузка файла в S3 backet
    aws --endpoint-url=http://localhost:4566 s3 cp hello.txt s3://local-test-bucket/

3️⃣ Проверим список файлов в баскете
    aws --endpoint-url=http://localhost:4566 s3 ls s3://local-test-bucket/

4️⃣ Деплой локальной Lambda
    aws --endpoint-url=http://localhost:4566 lambda create-function \
    --function-name my-local-lambda \                                   /* Дает имя твоей функции (можешь назвать как угодно)*/
    --runtime nodejs18.x \                                              /* Говорит AWS, что мы используем Node.js (версия 18.x отлично подходит для скомпилированного TS).*/            
    --handler index.handler \                                           /* Указывает, что точку входа нужно искать в файле index.js в функции handler.*/                                       
    --role arn:aws:iam::000000000000:role/lambda-role \                 /* В настоящем AWS здесь был бы сложный идентификатор прав доступа. В LocalStack мы можем передать любую фейковую строку, он не проверяет права так строго. */
    --zip-file fileb://function.zip                                     /* Передает сам архив с кодом.*/

5️⃣ Запуск нагей локальной функции
    aws --endpoint-url=http://localhost:4566 lambda invoke \
    --function-name my-local-lambda \
    output.txt 




1. Работа с образами (Images)
Образ — это "чертеж" или шаблон твоего приложения.
        docker pull <image_name> — скачать образ из Docker Hub.
    Пример: docker pull nginx (скачать официальный образ веб-сервера Nginx).

    docker images (или docker image ls) — посмотреть список всех скачанных образов на твоем ПК.
    docker build -t <name> . — собрать свой собственный образ из Dockerfile.
Пример: docker build -t my-app . (собрать образ с тегом "my-app" из текущей папки).

    docker rmi <image_id> — удалить образ.

2. Управление контейнерами (Containers)
Контейнер — это запущенный экземпляр образа (то, что реально работает).
        docker run <flags> <image> — создать и запустить контейнер.
    Пример: docker run -d -p 8080:80 --name web nginx

-d — запустить в фоновом режиме (detached).
-p 8080:80 — пробросить порт 8080 с твоего ПК на порт 80 внутри контейнера.
--name web — дать контейнеру понятное имя.
docker ps — список всех запущенных контейнеров.
docker ps -a — список вообще всех контейнеров (включая остановленные).
docker stop <name_or_id> — остановить контейнер.
docker start <name_or_id> — запустить ранее остановленный контейнер.
docker rm <name_or_id> — удалить контейнер.
docker exec -it <name> bash — "войти" внутрь работающего контейнера в терминал.

3. Очистка системы (Pruning)
Команды для удаления мусора, которые мы обсуждали в тесте.
docker system prune — удалить все остановленные контейнеры, неиспользуемые сети и "висячие" (dangling) образы.
docker container prune — удалить только остановленные контейнеры.
docker image prune — удалить только неиспользуемые образы.

4. Примеры использования в реальных задачах
Пример №1: Запуск базы данных PostgreSQL
Если тебе нужна база данных для разработки, не нужно её устанавливать в систему:

Bash
docker run --name my-db -e POSTGRES_PASSWORD=mysecret -d -p 5432:5432 postgres
Здесь -e задает переменную окружения (пароль).

Пример №2: Проверка ресурсов (Shared Memory)
Помнишь вопрос про 64 МБ? Вот как проверить это на практике:

Bash
docker run --rm alpine df -h /dev/shm
Эта команда запустит временный контейнер (--rm удалит его сразу после выполнения), покажет размер памяти и закроется.

Пример №3: Увеличение Shared Memory при запуске
Если твоему приложению (например, Chrome в Docker) мало 64 МБ:

Bash
docker run -d --shm-size=256m my-browser-app
Полезный "чит-код"
Если контейнеров и образов стало слишком много и они занимают всё место на диске, самая мощная команда для очистки:

Bash
docker system prune -a --volumes
Внимание: это удалит ВООБЩЕ ВСЁ, кроме запущенных контейнеров (включая все скачанные образы и данные в томах).
