1. Что такое PostgreSQL и почему такое название?
PostgreSQL — это самая мощная и популярная в мире база данных с открытым исходным кодом. Представь PostgreSQL как очень строгий и надежный Excel, который может хранить миллиарды строк. Он умеет следить за тем, чтобы данные были целостными, умеет быстро находить нужную информацию среди миллионов записей и, главное, позволяет сотням пользователей одновременно читать и писать данные, не ломая их.
Прокси которые можна использовать для связки с PostgreSQL могут быть:
            Vercel Postgres,
            Supabase,
            MySQL

Немного истории:
      - В 80-х годах в Калифорнийском университете в Беркли существовала база данных под названием Ingres (INteractive Graphics and REtrieval System).
      - Когда создали ее улучшенную версию, которая поддерживала более сложные запросы (объектно-реляционную модель), её назвали Post-Ingres (то есть "после Ingres"). Со временем название сократилось до Postgres.
      - SQL — это аббревиатура Structured Query Language (язык структурированных запросов), стандарт для общения с реляционными базами. Поэтому в 90-х добавили "SQL", чтобы подчеркнуть, что это база данных, работающая по стандартам.

2. Алгоритм: Как подключать новую БД (Supabase)

Шаг 1: Проектирование схемы (SQL Editor)
Не пытайся делать это "на лету" в коде. Сначала опиши структуру таблиц.

Создание:

SQL:
    CREATE table allowed_reviewers(    // название таблицы
      id serial primary key,           // id с уникальным идентификатором для каждой строки
      phone varchar(13) unique not null, // Поскольку номер телефона содержит знак + пожтому он уже не может быть целым числом. UNIQUE для телефона — это не даст тебе случайно добавить одного и того же человека в базу дважды.
      costumer_name varchar(50) not null
    );
При создании таблички выбираем безопасность (RLS):

Ещё один пример для создания таблички в котором будт храниться массивы (например массив ссылок на фотографии):
        CREATE TABLE reviews (
          id SERIAL PRIMARY KEY,
          -- Связываем отзыв с таблицей допусков (чтобы знать, кто именно оставил)
          reviewer_id INT REFERENCES allowed_reviewers(id) ON DELETE SET NULL, 
          
          -- Имя, под которым отзыв будет отображаться на сайте
          display_name VARCHAR(50) NOT NULL, 
          
          -- Текст отзыва
          review_text TEXT NOT NULL,
          
          -- Рейтинг (ограничиваем строго от 1 до 5)
          rating INT NOT NULL CHECK (rating >= 1 AND rating <= 5),
          
          -- Ответ Марианны (может быть пустым)
          admin_reply TEXT,
          
          -- МАССИВ ССЫЛОК НА ФОТОГРАФИИ (Вот магия для нескольких фото!)
          photos TEXT[] DEFAULT '{}',
          
          -- Статус модерации: 'pending' (на проверке), 'approved' (одобрен), 'rejected' (отклонен)
          status VARCHAR(20) DEFAULT 'pending', Это критически важно! Так как клиенты могут загружать фото, ты не должен выводить отзывы на сайт сразу после отправки. Вдруг кто-то загрузит туда что-то непотребное? Отзыв сохраняется со статусом pending (ожидает). Марианна в админке смотрит фото, читает текст и меняет статус на approved (одобрено). И только отзывы со статусом approved рендерятся на сайте.
          
          -- Дата и время создания отзыва
          created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
        );

Поскольку во втором варианте мы хотим использовать фото, тогда нам нужно создать Storage Bucket. В бесплатном тарифе Supabase предоставляется 1 ГБ свободного места для хранения файлов (включая фото). При среднем размере фото 1 МБ — около 1 000 фото.
    Перейди в раздел Storage:
        - В левом боковом меню твоего дашборда Supabase найди иконку папки — она называется Storage.
        - Создай новый Bucket. Нажми на зеленую кнопку "New bucket" (Новый бакет).
        - Настрой параметры бакета:
              - Name (Имя): Напиши review-images (используй только маленькие буквы и дефис).
              - Public bucket (Публичный): Обязательно включи этот тумблер! Это критически важно. Если он будет выключен, посетители сайта не смогут увидеть фотографии в отзывах, так как браузер не получит к ним доступ.
              - Остальные настройки (File size limits, Allowed MIME types) можешь пока оставить по умолчанию или, если хочешь перестраховаться, в "Allowed MIME types" впиши image/*, чтобы загружать можно было только картинки.
        - Нажми "Create".
        - Проверь настройки: Теперь в списке слева появится твой бакет review-images, и рядом с его названием должен быть значок открытого замочка (или надпись "Public").


SQL
ALTER TABLE table_name ENABLE ROW LEVEL SECURITY;
CREATE POLICY "Public can view" ON table_name FOR SELECT USING (true);
CREATE POLICY "Public can insert" ON table_name FOR INSERT WITH CHECK (true);
Шаг 2: Настройка клиента (React)
Установи: npm install @supabase/supabase-js.

Создай supabaseClient.js (если еще нет).

Важно: Убедись, что ключи в .env соответствуют твоему сборщику (VITE_ для Vite или REACT_APP_ для CRA).

Шаг 3: Реализация CRUD (Frontend)
Все операции с данными строятся по шаблону:

Чтение (Read):

JavaScript
const { data, error } = await supabase
  .from('table_name')
  .select('*')
  .eq('column', value) // Фильтр
  .order('id', { ascending: false }); // Сортировка
Запись (Insert):

JavaScript
const { data, error } = await supabase
  .from('table_name')
  .insert([{ column: value }]);
Обновление (Update):

JavaScript
const { data, error } = await supabase
  .from('table_name')
  .update({ column: new_value })
  .eq('id', target_id);
Шаг 4: Debugging (если что-то не работает)
Если запрос не проходит, действуй по алгоритму:

Консоль: Что пишет error? (Ошибка 409, 403, 401?).

Network: Открой вкладку "Network" в браузере, найди красный запрос к Supabase, посмотри Response — там всегда написано, почему база отказала (например, "row violates row-level security policy").

Таблица: Проверь, создана ли таблица и нет ли опечаток в названии колонок.

Совет на будущее:
Когда у тебя появится задача, например, "сделать личный кабинет для ученика", ты просто создаешь таблицу users_courses, прописываешь RLS политику (чтобы пользователь видел только свои данные), и используешь этот же шаблон.
