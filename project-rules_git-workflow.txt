После создание нового проекта, особенно если вы работаете в команде, необходимо выстроить правильный и безопасный процесс (Workflow), для этого нужно настроить три вещи:
    - защиту в GitHub;
    - структуру веток;
    - правила хостинга.

1️⃣ Защита продакшена в GitHub (Branch Protection)
        Чтобы никто (включая тебя самого по ошибке) не мог сломать продакшн, ветку main нужно заблокировать для прямой записи.
            ➡️ В репозитории GitHub перейди во вкладку Settings -> Branches:
                    - Add classic branch protection rule. Это проверенный и самый простой способ защитить ветку для небольшого проекта;
                    - Add branch ruleset — это новая, более сложная система гибких правил, которую GitHub сделал для больших корпоративных команд.
                Нам нужно Add classic branch protection rule -> В самом верхнем поле Branch name pattern впиши main -> Поставь галочку возле Require a pull request before merging -> Если хочешь защитить продакшн от самого себя (чтобы по привычке случайно не прописать git push origin main), найди ниже настройку Do not allow bypassing the above settings и тоже включи её. Это заставит даже администратора (тебя) следовать правилам и делать Pull Request -> Create.
                ⚠️ Your protected branch rules for your branch won't be enforced on this private repository until you move to a GitHub Team or Enterprise organization account. - Такое сообщение может появиться в том случаи если наш репозиторий приватный (Private), и мы, используем бесплатный тариф GitHub. Политика GitHub такова: автоматическая блокировка веток бесплатно работает только для публичных (Public) репозиториев. Для приватных они требуют платную подписку.То есть сейчас система не ударит тебя по рукам, если ты по привычке сделаешь git push origin main.
                Есть три варианта решение этой проблемы:
                    - Мы оставляем репозиторий приватным. Защиты от «случайного пуша» от GitHub не будет, но мы всё равно выстраиваем правильный процесс. Мы создаем ветку develop и просто договариваемся (ты сам с собой и с будущими коллегами): «Мы пишем код только в ветках от develop, а в main переносим изменения только через создание Pull Request».
                    - Сделать репозиторий Public: Если твой код не содержит секретов, ключей от API и не является строгой коммерческой тайной, ты можешь зайти в Settings -> General и в самом низу изменить видимость репозитория на Public. Как только ты это сделаешь, защита main мгновенно заработает бесплатно, но гаш код ьудет доступен всем!
                    - Заплатить GitHub.

2️⃣ Новая структура веток (Git Branching)
    Тебе нужно уйти от работы напрямую в ветке main.
        - main (Production): Это «священная» ветка. Здесь лежит только стабильный, протестированный код, который видят пользователи твоего сайта;
        - develop (Development): Это ветка для разработки. Сюда попадает свежий код, который нужно протестировать перед тем, как отдать пользователям;
        - Feature-ветки: Если ты или твой коллега начинаете делать большую задачу (например, новый модуль оплаты), вы создаете отдельную ветку от develop (например, feature/payment-module), спокойно пушите туда незаконченный код каждый день, и это ничего не ломает;
            ➡️ Terminal -> nаш проект на локале -> git checkout main -> git pull
                ⚠️ Возможно что терминал потребовал добавить токен:
                    Создаем токен на GitHub:
                        - Зайди на GitHub, нажми на свою аватарку в правом верхнем углу и выбери Settings (Настройки).
                        - Прокрути левое меню в самый низ и нажми Developer settings.
                        - В левом меню выбери Personal access tokens -> Tokens (classic).
                        - Нажми кнопку Generate new token (справа сверху) -> Generate new token (classic).
                        - В поле Note напиши любое имя (например, "Laptop WSL").
                        - В разделе Select scopes поставь галочку возле repo (это даст токену права на чтение и запись кода). Остальные галочки можно не трогать.
                        - Прокрути вниз и нажми зеленую кнопку Generate token.
                        ‼️Важно: Сразу скопируй появившийся длинный код! GitHub покажет его только один раз. Это и есть твой новый "пароль" для терминала.
            ➡️ Продолжаем работу в терминале: git checkout -b develop
                                               git add .
                                               git commit -m "Move current work to develop branch"
                                               git push -u origin develop
        Отлично, мы создали двк ветки в нашем проекте:
            - main — это витрина для пользователей;
            - develop — твоя рабочая мастерская.
        Допустим, ты неделю работал в ветке develop, сделал там кучу коммитов, всё проверил локально — всё работает супер. Пришло время показать это пользователям.
        Твои действия:
            - Ты заходишь на сайт GitHub.
            - Нажимаешь кнопку New pull request.
            - Указываешь направление: из develop в main (то есть base: main <- compare: develop).
            - Нажимаешь Create pull request и затем Merge pull request.
            Всё! Твой идеальный код перелился в main, и хостинг (Netlify) автоматически обновляет боевой сайт.
        
        Что делать, если нужна новая (ещё одна) ветка для какой-то задачи? Это самый важный момент командной работы (или правильной работы в одиночку).
            - Новые ветки для задач (их называют feature-ветки) всегда создаются от develop и возвращаются обратно в develop. Они НИКОГДА не идут напрямую в main.
              Как выглядит полный цикл работы над новой задачей:
                - Старт: Ты находишься в ветке develop. Тебе нужно сделать новую страницу контактов. Ты создаешь от неё ветку:
                  git checkout -b feature/contacts
                - Работа: Пишешь код, делаешь коммиты, пушишь эту ветку на GitHub. Продакшн (main) при этом в безопасности, develop тоже пока не тронут.
                - Завершение задачи: Ты закончил страницу контактов. Идешь на GitHub и делаешь Pull Request из feature/contacts в develop.
                - Слияние: Ты (или твой коллега) проверяете код, нажимаете Merge. Теперь страница контактов появилась в develop.
                - Публикация: Когда в develop накопится достаточно таких проверенных задач (например, страница контактов и новый футер), ты делаешь один общий Pull Request из develop в main, как мы обсуждали в первом пункте.

3️⃣ Наша цель сейчас: сделать так, чтобы основной домен обновлялся только при попадании кода в main, а для ветки develop генерировалась отдельная секретная ссылка, где ты сможешь всё покликать и проверить перед релизом.
    ➡️ Настраиваем основную ветку (Production):
        - Зайди в свой аккаунт на сайте Netlify и выбери свой проект (ited).
        - Перейди в раздел Site configuration (слева) -> Build & deploy.
        - Прокрути вниз до секции Branches and deploy contexts.
        - Найди пункт Production branch. Там должно быть написано main. Если вдруг там написано что-то другое, нажми Configure и впиши main.
        ‼️Теперь Netlify знает, что боевой сайт собирается только отсюда.

    ➡️ Включаем тестовую ссылку для develop. В том же самом разделе Branches and deploy contexts, чуть ниже:
        - Найди пункт Branch deploys.
        - Нажми Configure (или Edit).
        - Выбери вариант Let me add individual branches (Разрешить мне добавить отдельные ветки).
        - В появившемся поле впиши название нашей рабочей ветки: develop.
        - Сохрани изменения (Save).
    
    ➡️ Проверяем Deploy Previews (Превью для Pull Request)
        - В этом же меню убедись, что в разделе Deploy Previews стоит галочка Any pull request against your production branch (или нечто похожее, обычно включено по умолчанию).
        - Это та самая магия, которая будет создавать временный сайт, когда ты нажмешь "New Pull Request" на GitHub.

4️⃣ После того как наша ветка develop прошла успешно все тесты нам ее нужно спарить с нашей главной веткой.
Спаривание (размещение) ветки develop на main:
 - GitHub -> наш репозиторий -> скорее всего, сразу увидишь плашку "Compare & pull request" для ветки develop. Смело нажимай на неё! (Если её вдруг нет, перейди на вкладку "Pull requests", нажми "New pull request" и выбери направление: base: main <- compare: develop).
 В открытом окне создаём Pull requests, записываем название, коротко о чём это и нажимаем Create pull request. После того как деплой прошёл успешно и конфликтов не выявлено смело нажимаем Merge pull request -> оставляем коментарий, если нужно -> Config merge
 Как только ты нажал Merge, GitHub дал сигнал хостингу. Можешь открыть панель Netlify — ты увидишь, как там запустилась новая сборка с пометкой Production (или main).
            

Алгоритм действий при создании нового проекта:
 - создаём репозиторий и клонируем его на локальный ПК. Или создаём проект локально, инициализтркем его через какие-то фреймворки, чистим архитектуру и пушим на GitHub;
 - локально добавляем ветку develop (git checkout -b develop) и отправляем назад на GitHub (git push origin develop);
 - Делаем baseline (фундамент) сайта на ветке develop (настраиваем архитектуру, титул, favicon, Readme.md ....), по результатам пушим на GitHub.
 - Pull Request (PR) на GitHub: Настрой направление слияния:
                                    - base: main
                                    - compare: develop
    -> Нажми Create pull request. В описании можешь написать «Настройка базовой архитектуры проекта».-> Нажми Merge pull request и затем Confirm merge. Теперь твой код в облаке (в ветке main) обновился. Нам нужно подтянуть эти изменения на твой компьютер, чтобы локальный main тоже знал об архитектуре-> VScode git checkout main -> git pull origin main
- Посде того как в main подтянули изменения, переходим в ветку deploy и там создаём новую например: git checkout -b feature/layout-header и начинаем работать над проектом.
- После того как наш код проверен и исправлены все замечания в самом низу нашей ветки с Pull request зелёная кнопка -> Merge pull request:
                    - Create a merge commit (Обычный мерж - по умолчанию), Git берет все твои коммиты (даже самые мелкие, вроде "поправил опечатку", "забыл запятую") и переносит их в master (main), создавая сверху еще один специальный «коммит слияния».
                        ➕ Плюсы: Видна вся история каждого твоего шага.
                        ➖ Минусы: История master превращается в "кашу" из сотен мелких промежуточных коммитов от разных разработчиков.
                    - Squash and merge (Сплющить и слить) 🏆 РЕКОМЕНДУЮ (В современной коммерческой разработке это самый частый стандарт для продуктовых фичей), Git берет все твои 5-10 коммитов, которые ты сделал за эти дни в ветке, "сплющивает" их в один единственный, большой и красивый коммит и кладет его в master (main).
                        ➕ Плюсы: Идеально чистая история в главной ветке. Никто не увидит твоих промежуточных мучений или коммитов вроде "fix texts.json". В истории master будет просто одна аккуратная запись: "CRMS-5: Сделана форма чекаута".
                        ➖ Минусы: Ты не сможешь откатить отдельную мелкую правку (только всю фичу целиком), но для продуктовой разработки это обычно и не нужно.
                    - Rebase and merge (Перестроить и слить), Git берет твои коммиты по одному и аккуратно переносит их в самый конец ветки master, выстраивая в ровную линию без «коммита слияния».
                        ➕ Плюсы: Ровная линейная история.
                        ➖ Минусы: Все твои мелкие промежуточные коммиты всё равно попадут в master.
                -> Confirm squash and merge.
                -> возвращаемся в VScode (наш проект) -> подтягиваем все изменеие в master (main) -> удаляем робочую ветку (она своё уже отработала, если нам что-то нужно будет доработать так создаём новую ветку и весь цикл разроботки повторяеться). Если Git будет ругаться, что ветка не полностью слита — такое часто бывает именно после Squash-мержа, так как хэши коммитов меняются. В этом случае просто используй заглавную -D для принудительного удаления.
- Рабочие гаши ветки на стадии разроботки или тестирования, мы после push на GitHub для качественного теста в реальной обстановки (это не продакшн жто отдельная ссылка на хостинге), мы должны ее соединить с веткой develop:
        - git checkout develop -> git pull origin develop (на всякий случай вдруг там не были последнии изменения) -> git merge <имя нашей робочей ветки> -> git push origin develop
